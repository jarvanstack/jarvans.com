# 作为程序员，我们应该更关注代码质量还是只需要以完成功能就好了？

## 1. 反向思考，什么不行的

> 芒格: 如果我知道我会死在那，我就不会去

- 开发的太慢: 资源耗尽，市场被抢走, 产品会死
- Bug过多: 用户体验过差，产品会死
- 性能过差: 上量就炸服，用户没法用，产品会死
- 可维护性过差:后面加功能，改功能，成本非常高，产品会慢慢死

## 2. 产品关心 vs 技术关心

* 产品关心: 开发快, Bug 少
* 技术关心: 性能, 可维护性

## 3. 这4个都想要，不可能的

很明显, 这是一个 trade-off 问题, 我们需要在产品需求和代码质量之间找到一个平衡点。

<br/>

想要做到这个, 仅仅会技术是不够的. **要理解上下文，就要有大局观，了解业务和产品，还要了解当前的形势**。

<br/>

我知道这很难，因为公司都倾向于把人变成螺丝钉，让大家不要去关心自己工作以外的大局，但想要做到四级工程师, 这是必须做好的一件事情。


<br/>

> 一个好的工程师能知道现在有的资源是多少，我们是应该造一个快速过河的浮桥，还是坚固一些的石桥，还是能用百年的海湾大桥




## 4. 故事 1

创游是一个游戏 UGC 平台, 全球翻译用的机翻, 产品说效果不好, 新的游戏改用大模型翻译, 开发搭建了一个工作流来做翻译, 因为游戏比较复杂, 产品想要上线

他来问 "怎么上线?"

背景: 我们后端代码都是跑 k8s 的, golang, 微服务. 他用的 Python 跟我们任何基建都不兼容

我说 "不用上线"

"不用上线怎么可能? 产品说要上线"

"我们 80% 的用户都玩这 3 款游戏, 对吧 ?"

"对"

"你把这 3 款游戏在你本地翻译了, 然后把数据给我不就行了, 然后因为是真实线上游戏, 一定有大量反馈, 然后你再优化工作流, 因为没有环境什么比本地迭代更快"
