# 以终为始

> 核心观点：需求只是实现价值的一种假设，程序员应该对结果负责，而不是只对需求负责。

## 建议结构

1. 从“需求全部按时完成，但没人使用”开场。
2. 区分需求、产出、结果和价值。
3. 写清接到任务后必须追问的三件事：为谁解决、解决什么、如何验证。
4. 给出一次因为理解终点而删除或改变需求的案例。
5. 讨论边界：什么时候应该执行，什么时候必须挑战需求。

## 待补材料

- [ ] 一个无效交付的成本数字
- [ ] 一个从业务指标倒推技术方案的例子
- [ ] 上线前后的验证指标
- [ ] 与产品经理意见不一致时的沟通方式

## 可用检查项

- 用户是谁？
- 用户行为会发生什么变化？
- 不做会怎样？
- 上线后用什么数据判断有效？
