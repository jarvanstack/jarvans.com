# 别急着写代码：把马斯克的五步工作法用在软件开发中

周一上午，刚入职半年的程序员小林收到一个需求：

> 两周内做一个“经营周报平台”。要有二十项指标、趋势图、筛选器、Excel 和 PDF 导出、自定义报表、订阅、定时邮件、历史版本、手机适配，还要有独立的权限系统。

小林把需求拆成数据库表、接口和页面，排期正好十天。第九天，产品经理来看演示，才发现大家真正想解决的问题是：**每周一开会前，六位区域经理需要快速找出销售额明显下跌的门店。**

二十项指标中，会上只看四项；没人需要自定义报表；公司已有权限系统；“退款率”还因为统计口径不同，整张报表都算错了。

小林确实写了很多代码，却没有更接近问题的答案。

这是一个虚构的场景，但类似的事几乎每天都在软件团队里发生。我们把需求文档当成答案，把完成工单当成价值，最后正确地实现了一个错误、过大或者过早的方案。

**程序员的价值，不取决于完成了多少需求，而取决于用多合理的成本，解决了多少真实问题。**

马斯克用于工程与制造的五步工作法，恰好提供了一套检查顺序：

1. 质疑每一项要求
2. 删除不必要的部分或流程
3. 简化和优化剩余部分
4. 加快周期
5. 自动化

这套方法最重要的不是五个动词，而是它们的顺序。先后顺序一旦反了，你可能会把一个不该存在的功能做得非常精美，把一条错误的流程跑得非常快，再用自动化把浪费放大。

## 第一步：质疑要求，而不是质疑人

拿到需求后，初级程序员最常问的是：“什么时候要？”

更好的第一个问题是：“我们想解决什么问题？”

“运营部需要一个周报平台”不是问题，只是一个带着方案的要求。继续追问后，小林得到了一组更具体的信息：

- 谁会使用：六位区域经理
- 在什么场景使用：每周一的经营会议
- 现在有什么困难：运营要花两个小时拼 Excel，经理还要逐行寻找异常门店
- 哪些信息真正影响决策：销售额、订单量、退款率、客单价，以及它们的周环比变化
- 怎样算成功：经理能在三分钟内找出下降超过 15% 的门店

到这里，任务已经从“开发一个功能齐全的平台”变成了“帮助区域经理在三分钟内发现异常门店”。前者规定了手段，后者说明了结果。

马斯克强调，每项要求都应该能追溯到一个具体的人，而不是一个模糊的“业务部门”“法务”或“领导”。这不是为了追责，而是因为只有找到了解上下文的人，团队才能继续确认：

- 这是事实，还是猜测？
- 这个限制现在仍然存在吗？
- 不做会发生什么？
- 有没有数据、用户反馈或法规条文支持它？
- 什么结果能证明它有效？

质疑需求也不等于对产品经理说“这个需求很蠢”。需求在传递过程中必然会丢失信息，产品经理和程序员都可能提出错误方案。专业的表达应该针对假设，而不是针对人：

> 我先复述一下目标：区域经理需要在周一开会前找出异常门店。如果第一版只提供四项核心指标和周环比，先让六位经理试用，能不能验证这个目标？PDF 和自定义报表有谁会在什么场景使用？

这样的质疑不是抬杠，而是在避免整个团队一起努力地走错路。

当然，质疑之后也可能发现需求完全合理。五步工作法不是“凡事反对”，而是**把每项要求暂时当成一个需要验证的假设**。

## 第二步：删除，但要留下最小闭环

问题确认之后，不要马上想“怎么实现全部功能”，先问：“哪些东西不做，也能解决这个问题？”

小林和产品经理重新检查了十项功能：

| 原计划 | 第一版的决定 | 理由 |
| --- | --- | --- |
| 二十项指标 | 缩减为四项 | 只有四项会触发会议决策 |
| 趋势图 | 删除 | 排序和周环比已能暴露异常 |
| 多条件筛选 | 只保留周次和区域 | 六位用户只有这两个查询场景 |
| Excel、PDF 导出 | 删除 | 经理在线查看即可 |
| 自定义报表 | 删除 | 暂无第二种报表需求 |
| 报表订阅 | 删除 | 第一版先人工发送链接 |
| 定时邮件 | 删除 | 流程还没有验证，不急着自动化 |
| 历史版本 | 删除 | 现有数据库已经保留历史数据 |
| 手机适配 | 延后 | 会议中统一使用电脑 |
| 独立权限系统 | 删除 | 复用内部管理后台的权限 |

删完以后，第一版只剩下一个很小的闭环：

> 区域经理登录现有管理后台，选择周次和区域，看到按跌幅排序的门店，以及四项指标的周环比。

用户能完成任务，团队也能观察结果，所以它是一个闭环。只有一张不能登录、没有真实数据的静态页面，不是闭环；只写查询接口却没有人使用，也不是闭环。

删除并不意味着永远不做。它的含义是：**在证据出现之前，先不支付开发和维护成本。** 如果用户试用后确实需要 PDF，就把 PDF 加回来。马斯克甚至用“如果没有加回大约 10%，说明删得还不够”来提醒团队真正做取舍，而不是象征性地砍两个边角功能。

在软件开发中，可以删除的不只是页面功能，还包括：

- 没有消费者的接口和字段
- 只有一次使用场景的“通用”框架
- 过早拆分的微服务
- 重复的审批、会议和手工交接
- 为假想流量准备的缓存、消息队列和分库分表

但不要把安全、数据正确性、必要的错误处理和合规要求当成“非核心功能”。最小闭环可以小，不能残缺。**删功能，不删工程底线。**

## 第三步：简化和优化，不是炫技

删到不能再删后，才轮到讨论怎么实现。

针对剩下的周报需求，小林列出三个方案：

| 方案 | 做法 | 预计时间 | 代价 |
| --- | --- | ---: | --- |
| A | 新建报表微服务、独立数据库和前端项目 | 8 天 | 边界清晰，但第一版成本最高 |
| B | 开发可配置的通用报表引擎 | 5 天 | 看似灵活，但需求只有一种 |
| C | 在现有管理后台增加一个查询和一个页面 | 2 天 | 扩展性有限，但足以验证当前问题 |

团队选择了 C。它复用现有登录权限和部署流程，用一条聚合查询计算四项指标，再在现有后台增加一个页面。只有当第二、第三种报表真的出现时，团队才考虑抽象公共能力。

这就是软件里的简化：减少组件、概念、状态和交接，让一个新人也能顺着代码找到数据从哪里来、在哪里计算、最后显示到哪里。

优化则需要证据。假如查询只处理一周的数据，响应时间是 300 毫秒，就不需要先加 Redis 和消息队列。如果真实数据下响应时间超过两秒，再通过执行计划确认是缺少索引、扫描范围过大，还是计算方式有问题，然后处理真正的瓶颈。

所以顺序必须是：

> 先确认它应该存在，再删掉不需要的部分，然后让剩下的方案更简单，最后才优化被测量出来的瓶颈。

有时一个简单方案看起来像“曲线救国”：不做独立平台，而是在现有后台增加一页；不做实时计算，而是接受每天更新一次。只要它满足用户目标和明确的约束，这不是偷懒，而是重新打开了解题空间。

但如果方案只是把复杂度转嫁给用户，例如要求运营每天手工修改数据库，那只是隐藏成本，不是真正的简化。

## 第四步：加快反馈，不是催大家加班

小林原来的计划是十天后一次性交付。新计划变成了：

- 两小时后：用四行样例数据画出页面草图，确认信息是否够用
- 第一天：接入一个区域的真实数据，和运营核对指标口径
- 第二天：部署到测试环境，走通登录、查询和异常状态
- 第三天：向六位区域经理开放，观察他们能否在三分钟内找到异常门店

第一次看真实数据时，产品经理发现“退款率”有两种定义：按申请退款时间统计，或按退款成功时间统计。如果等到第十天才发现，接口、测试和页面说明都要返工；第一天发现，可能只需要改一条查询和一行文案。

所以，你关于“尽早让产品经理看到效果”的理解是对的。但第四步不只是更早演示，它的目标是缩短整个反馈周期：

> 想法 → 实现 → 测试 → 交付 → 用户反馈 → 下一次修改

常见做法包括：

- 按可验证的用户结果切分任务，而不是先做完所有数据库，再做所有接口
- 尽早使用真实数据和真实权限，先暴露最危险的假设
- 先向少量用户开放，控制错误的影响范围
- 复用团队已有的自动化测试、构建和发布能力
- 让每次改动都足够小，小到可以快速验证和回退

需求不一定每次都会改，但软件开发本身包含探索。越早拿到证据，改变方向的成本通常越低。

加速也不是跳过测试、压缩思考时间或者让程序员长期加班。那是在提高打字速度，同时积累缺陷和返工。真正应该提高的是**单位时间内获得有效反馈的速度**。

## 第五步：最后自动化，让正确的事自动发生

第一版上线后，团队连续观察了三个周一：

- 六位区域经理都在使用
- 四项指标的定义没有再变化
- 他们确实能在三分钟内找到异常门店
- 运营每周仍要手工打开页面并发送链接

这时，团队才把最后一个重复步骤自动化：每周一早上生成报表，完成数据完整性检查后发送链接；失败时告警并允许人工重跑。

为什么自动化放在最后？因为自动化会忠实地重复当前流程。流程错了，它就稳定地犯错；需求还在变化，它会增加修改成本；任务根本不重要，它只会让浪费运行得更快。

自动化之前，至少确认两件事：

1. 这件事已经重复发生，而且过程基本正确、相对稳定。
2. 节省的时间和减少的错误，大于开发、监控和维护自动化的成本。

“自动化会降低灵活性”只说对了一部分。为一条尚未稳定的业务流程开发专用自动化，确实可能固化错误假设；但成熟的自动化测试、持续集成和一键回滚，反而会提高软件的可变更性。第五步不是要求前三步全部手工完成，也不是让团队最后才写测试，而是提醒我们：**不要急着为一个还不确定是否正确的流程，建造昂贵的专用自动化。**

好的自动化还应该保留刹车和仪表盘：可以暂停、可以人工重跑、失败会告警、结果能够核对。否则，团队只是把手工错误变成了无人发现的自动错误。

## 五个最常见的误用

| 误用 | 更准确的理解 |
| --- | --- |
| 质疑需求就是挑战产品经理 | 共同验证问题、证据和约束，不攻击提出要求的人 |
| 删除就是做一个粗糙 Demo | 交付最小但完整的用户闭环，质量底线仍然存在 |
| 优化就是提高代码性能 | 先降低方案和系统复杂度，再处理测量到的瓶颈 |
| 加速就是赶工和加班 | 缩小批次，尽早验证，缩短从想法到反馈的周期 |
| 自动化必须等所有功能完成 | 新流程稳定后再做专用自动化，已有的测试和发布能力照常使用 |

## 初级程序员可以直接使用的五个问题

下一次接到需求，不需要召开一场“五步工作法研讨会”。开工前花十分钟，依次问完下面五个问题就够了：

1. **质疑：** 谁遇到了什么问题？有什么证据？什么结果算成功？
2. **删除：** 哪些功能暂时不做，用户仍然能完成最核心的任务？
3. **简化：** 能否复用现有系统？有没有组件更少、概念更少的方案？
4. **加速：** 最快什么时候能让真实用户看到真实结果？最大的未知能否今天验证？
5. **自动化：** 哪个重复、稳定并且容易出错的步骤，已经值得交给程序执行？

你也可以把它压缩成一句话，放在需求评审或代码评审前提醒自己：

> 它为什么要做？什么可以不做？怎样做得更简单？怎样更早得到反馈？最后，什么值得自动化？

## 最后：代码不是产量，问题才是

如果小林直接接受最初的清单，他会完成十项功能，十天后才知道方向有误。按照五步工作法，他用两小时发现认知偏差，删除八项未经验证的功能，用两天完成最简单的闭环，第三天获得真实反馈，三周后才自动化已经稳定的流程。

这里的数字只是为了说明方法，并不保证所有项目都能缩短到三天。医疗、支付、安全等高风险系统还需要更严格的验证，不能借“删除”和“加速”绕过必要控制。很小且可逆的改动，也不值得把五步变成一套沉重仪式。

但无论任务大小，这个顺序都值得在脑中走一遍。

初级程序员常用“我写了多少代码”证明自己；成熟的软件工程师更关心“我避免了多少错误的代码，并让真正的问题更早得到解决”。

**最快的开发，不是更快地写完所有需求，而是尽早发现哪些需求根本不必写。**
