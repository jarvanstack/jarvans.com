# 代码写完了，为什么产品经理说还没做完？

星期一，产品经理问程序员：“这个功能多久能做完？”

程序员看了一下需求，说：“两天。”

星期三，产品经理过来问：“今天可以全量上线吗？”

程序员很惊讶：“怎么可能？我只是刚把代码写完，还没有提测。”

测试的时候又发现两个 Bug，修完以后错过了发布窗口。等到灰度、全量上线，已经是下周一了。

产品经理觉得程序员延期了，程序员却觉得自己很准：说两天开发完，确实两天开发完。

两个人都没有撒谎，只是他们说的“做完”，根本不是同一件事。

**程序员回答的是还要写几天代码，产品经理问的是还要多久用户才能用上。**

## 1. 代码写完，不等于需求完成

一个软件功能从开始开发，到真正产生价值，通常要经过这些步骤：

```text
本地开发和自测
    ↓
测试环境验证
    ↓
正式环境验证和灰度
    ↓
全量上线
    ↓
观察用户反馈和数据
    ↓
确认用户的问题被解决
```

程序员很容易走到第一步，就说：“我做完了。”

但是在用户眼里，本地代码写得再漂亮，测试环境跑得再稳定，都没有任何区别。因为他还没有看到，也没有用上。

这就像一个厨师告诉客人菜十分钟做好。十分钟以后，他说：“菜已经切好了，炒和上桌还需要一些时间。”

厨师可能确实只用了十分钟切菜，但这不是客人理解的“做好”。客人关心的是多久能吃到。

产品经理也是一样。他当然会关心开发有没有风险，但真正能开始验证产品结果的时间，是功能全量上线以后。

所以当产品经理问“多久能做完”，只回答开发时间，给出的不是一个准确但不完整的答案，而是一个**无法帮助他做决定的答案**。

## 2. 需求单不是价值，只是一张路标

很多程序员会用这些东西证明自己做了很多工作：

* 这个月完成了 20 个需求；
* 写了 1 万行代码；
* 所有需求都按时提测；
* Jira 里面的任务都已经关闭。

这些可以说明工作量，但不能直接说明价值。

如果用户根本不需要这个功能，完成得越快，只是浪费得越快。如果功能已经上线，但是用户不会用、找不到入口，或者用了也没有解决问题，需求单关掉了，价值仍然没有产生。

上一篇文章[《程序员拿到需求，不要马上开始写代码》](/2-how-pro/five-steps-v2.md)讲了马斯克的五步工作法：先质疑需求，再删除、简化、加速，最后才自动化。

为什么第一步不是加快开发？

因为方向错了以后，速度越快，离目标越远。

**程序员产生的价值，是给用户创造的价值，不是完成了多少张需求单。**

需求单只是一张路标，提醒团队要解决哪个问题。把路标擦得很干净，不代表已经到达目的地。

## 3. “完成”至少有五种状态

团队应该把完成的定义（Definition of Done）提前说清楚。为了避免误解，不要只说一句“做完了”，而要说清楚现在到了哪一步。

| 状态 | 它真正代表什么 |
| --- | --- |
| 开发完成 | 代码已经写完，本地自测、联调通过，可以交给测试环境验证 |
| 测试完成 | 验收条件通过，已经具备发布条件 |
| 灰度完成 | 正式环境验证通过，少量用户可以使用，监控和回滚方案正常 |
| 上线完成 | 功能已经全量发布，所有目标用户都可以使用，系统运行稳定 |
| 价值完成 | 用户真的在使用，数据或反馈证明原来的问题被解决 |

这五种完成都可以成为一个阶段，但不能混在一起。

你可以说：“代码开发完成，今天下午提测。”

也可以说：“测试已经通过，明天灰度，预计周五全量。”

还可以说：“已经全量上线，接下来观察七天转化率。”

但是不要在代码只存在于自己电脑里时，对别人说整个需求已经完成。

**代码写完，只能叫代码写完。**

## 4. 用以终为始，倒推什么叫完成

以终为始，不是先想“我要写哪个接口”，而是先想“用户最后会得到什么”。

比如，产品经理说要给注册页面增加一个“一键登录”按钮。

如果从开发开始想，完成标准可能是：

> 按钮做好了，接口也调通了。

如果从用户价值开始想，完成标准会变成：

> 新用户可以更快地完成注册，注册成功率从 60% 提升到 70%。

这两个定义会带来完全不同的工作方式。

第二种定义会逼着团队继续思考：按钮用户能不能看到？失败以后提示是否清楚？老版本是否兼容？埋点有没有上报？上线以后看哪个数据？观察多久？如果注册成功率没有提升，谁来继续找原因？

一个需求开始以前，最好先回答四个问题：

1. 我们要为哪一类用户解决什么问题？
2. 什么现象或数据能证明问题被解决？
3. 功能什么时候全量上线，准备观察多久？
4. 如果结果不符合预期，谁来继续处理？

不是每一个需求都能直接换算成收入。

修复支付 Bug，价值可能是支付失败率下降；优化性能，价值可能是页面打开时间减少；做合规改造，价值可能是降低违规和停服风险。

指标可以不同，但必须知道自己想改变什么。

## 5. 估算的应该是上线时间，不只是编码时间

产品经理问“要多久”，通常是因为他要安排后面的事情：通知用户、准备运营活动、培训客服，或者向老板承诺发布时间。

这时他需要的不是一个孤立的开发工时，而是一个可以安排工作的日期。

不要只回答：

> 开发两天。

可以改成：

> 编码和自测需要两天，周三上午可以提测。测试预计一天，我预留半天修问题。顺利的话周五灰度，下周一全量。现在最大的风险是第三方登录还没有联调过。

这样的估算仍然可能变化，但它至少说明了：

* 每个阶段需要多久；
* 什么时候用户能用上；
* 中间依赖谁；
* 最大的不确定性是什么。

产品经理拿到这些信息，才能决定活动是周五开始，还是等到下周一。

**好的估算，不只是证明程序员没有延期，而是帮助整个团队安排下一步。**

如果测试由别的团队负责，或者发布窗口不由你决定，也不要假装这些时间不存在。你可以明确说明依赖和假设，但最终仍然要给出全量上线的预期时间。

## 6. 上线了，也可能没有完成

有些需求全量上线以后，团队就立刻庆祝，任务也被关闭了。

但是一周以后再看，可能只有三个人用过；也可能按钮点击率很高，真正完成操作的人却更少了；还可能新功能提升了注册率，却让投诉翻了一倍。

上线只代表我们把一个答案交给了用户，用户和数据才会告诉我们答得对不对。

当然，有些结果需要一个月甚至半年才能观察出来。任务不可能永远挂在“进行中”。

可以把执行状态和结果状态分开：

```text
执行状态：已经全量上线，系统运行稳定
结果状态：观察 14 天，注册成功率目标从 60% 提升到 70%
负责人：产品经理和开发共同复盘
```

工程任务可以在全量稳定以后结束，但产品需求还没有真正闭环。只有到了约定时间，大家一起看完结果，决定继续、调整还是下线，才算回答了最初的问题。

## 7. 一句可以直接使用的汇报模板

以后别人问“做完了吗”，可以这样回答：

> 当前已经完成【代码 / 提测 / 测试 / 灰度 / 全量】；预计【日期】全量上线；上线后观察【指标】到【日期】；目前最大的风险是【风险】，负责人是【姓名】。

这句话看起来比“做完了”麻烦一点，却可以少掉后面大量的误解。

团队也可以提前约定：默认说“完成”，究竟指开发完成、全量上线，还是价值验证完成。只要定义一致，很多所谓的延期和扯皮，本来就不会发生。

## 最后

初级程序员容易把自己的边界画在代码提交的地方：我写完了，后面是测试、运维和产品的事情。

成熟的工程师会继续往后看：测试能不能顺利验证，功能能不能安全上线，用户能不能正常使用，数据有没有变好。

他不一定亲自做完所有事情，但会知道整个结果现在在哪里，下一棒交给了谁，哪里可能让价值停下来。

**需求的终点不是代码仓库，也不是需求管理系统里的“已完成”，而是用户的问题真的被解决。**

所以，下一次再说“我做完了”以前，先问自己一句：

**用户已经得到价值了吗？**
