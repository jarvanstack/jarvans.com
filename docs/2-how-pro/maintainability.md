# 可维护性

> 核心观点：可维护性不是抽象的“代码优雅”，而是下一次修改能否被快速理解、局部完成并安全验证。

## 建议结构

1. 用一次小改动引发大面积修改的案例开场。
2. 从理解成本、修改范围、验证难度三个维度定义可维护性。
3. 选择两三个设计原则，用真实代码展示前后差异。
4. 解释“品味”如何通过命名、边界和删除不必要抽象体现。
5. 讨论边界：不要为想象中的未来过度设计。

## 待补材料

- [ ] 一段重构前后的代码
- [ ] 修改涉及的文件数、耗时或缺陷数
- [ ] 哪个抽象解决了真实变化，哪个抽象只是猜测
- [ ] Code Review 时判断可维护性的具体问题
