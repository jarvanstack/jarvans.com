# 程序员拿到需求，不要马上开始写代码

最让程序员难受的事情，不是代码写不出来。

而是辛苦做了一个星期，产品经理看了五分钟，说：“不对，不是这个意思。”

然后需求改了，代码也要跟着改。

我们很容易抱怨产品经理需求没想清楚。但反过来想，如果直到开发快结束，产品经理才第一次看到东西，那这个反馈确实来得太晚了。

我以前觉得，一个好的程序员就是开发快、Bug 少，别人提的需求都能按时完成。

后来我发现，**完成需求不等于创造价值。**

一个没有价值的需求，你做得越快，浪费得也越快。

马斯克有一个五步工作法：

1. 质疑每一项需求
2. 删除不必要的部分
3. 简化和优化
4. 加快迭代速度
5. 自动化

这五步最重要的是顺序。

## 第一步：先质疑需求

拿到需求，第一件事情不是想数据库怎么设计、接口怎么写，而是先问一句：

**我们到底想解决什么问题？**

比如产品经理说：“这个列表要加一个 Excel 导出。”

程序员通常会马上问：“要导出哪些字段？”

但你也可以先问：谁要用？导出来做什么？多久用一次？

你可能会发现，运营只是每个月要复制一次最近二十个订单号。那也许加一个“复制”按钮就够了，甚至现有功能已经可以解决。

当然，也可能问完以后，发现 Excel 导出确实要做。

质疑需求不是为了证明产品经理错了，也不是为了拒绝工作。需求通常只是别人先想到的一个解决方案，程序员要和他一起确认背后的问题。

**不要攻击提出需求的人，要质疑需求里面没有被验证的假设。**

## 第二步：删除不重要的功能

假设 Excel 导出确实要做。

讨论着讨论着，很容易又多出很多功能：可以选择字段、可以导出多种格式、可以查看导出历史、可以定时导出、还可以自动发邮件。

这些功能可能都有用，但不代表第一版都要做。

如果用户现在只需要把订单交给财务，第一版也许只需要一个固定格式的导出按钮。用户点一下，能拿到正确的文件，这件事就完成了。

删除不是这些功能永远不做，而是**没有证据证明它重要之前，先不做。**

马斯克说，如果删完以后，没有发现一些东西需要加回来，说明删得还不够。

因为大多数人所谓的“删除”，只是把十个功能删成九个。真正的删除，是继续问：没有这个功能，用户还能不能完成最重要的事情？

软件里面可以删除的不只是功能，也可能是一张表、一个微服务、一层没有必要的抽象，或者一次没有结论的会议。

最好维护的代码，就是没有写出来的代码。

## 第三步：简化和优化

删除之后，再想剩下的东西怎么做。

还是 Excel 导出。如果现在只有一个固定格式，就没有必要先做一个“通用报表平台”，也没有必要因为以后“可能有很多导出需求”，先设计一套复杂的配置系统。

在现有后台加一个接口，复用已有的查询和权限，可能已经足够。

如果数据量真的大到会超时，再考虑异步任务、消息队列和文件存储。如果现在导出只需要一秒，就不要先解决明年可能出现的问题。

简单不等于粗糙。

该有的权限、错误处理和数据准确性仍然要有。简化是减少没有必要的东西，不是把问题留给用户，也不是少做应该做的事情。

我们经常会碰到几个方案：A 最完整，但是要做五天；B 能解决大部分问题，只要两天；C 今天就能做，但是会留下明显的坑。

工程师要做的不是永远选择最简单的方案，而是根据现在的问题和资源，选择投入产出最合适的方案。

## 第四步：加快迭代速度

很多人把加速理解成加班，或者一天多写一点代码。

不是。

真正应该加快的是从“想到一个方案”到“拿到真实反馈”的速度。

做导出功能时，不需要等页面、权限、异步任务全部写完，再让产品经理验收。

第一天先导出二十条真实数据给他看。他可能马上会告诉你：这个不是用户看到的订单号，时间格式也不对，还少了一个财务必须要的字段。

这些问题第一天发现，改起来很便宜。开发结束以后才发现，前后端、测试和文档可能都要一起改。

产品经理看到真实的东西以后，确实更容易发现自己的想法不完整。这不一定是谁犯了错，而是很多问题只靠想象很难想清楚。

所以要把需求拆小一点，尽早使用真实数据，尽早让真正使用的人看到。

**速度不是单位时间写了多少代码，而是多久能知道自己做得对不对。**

## 第五步：最后再自动化

如果这个导出功能已经稳定下来，财务每周都要导出一次，而且每次的时间、格式和接收人都一样，这时可以把它自动化。

系统每周自动生成文件，检查成功后发给财务，失败了就报警。

但如果统计口径每周都在改，谁要接收也没确定，自动化只会让修改变得更麻烦。

自动化会提高效率，也会放大错误。一个错误的流程手工执行，可能只错一次；自动执行，可以稳定地错一百次。

所以自动化放在最后，不是因为自动化不好，而是应该先确认这件事情值得做，流程基本正确，而且会重复发生。

这也不代表写单元测试、使用持续集成都要等到最后。成熟的测试和发布工具反而会让修改更快。最后再自动化，主要指的是不要过早给一个还不稳定的业务流程造一套专用系统。

## 最后

以后拿到需求，可以先在脑子里问五个问题：

1. 这个需求到底要解决什么问题？
2. 哪些东西可以先不做？
3. 剩下的东西，能不能做得更简单？
4. 怎么尽早让用户看到，知道方向对不对？
5. 哪些已经稳定、重复的事情值得自动化？

不需要每次都开会，也不需要把它变成一个复杂的流程。小需求可能一分钟就能想完，大需求可能需要团队一起讨论。

初级程序员容易用“做了多少需求、写了多少代码”证明自己的价值。

但代码从来不只是成果，它也是以后要测试、修改和维护的成本。

**一个成熟的软件工程师，不只是能把代码写出来，也知道哪些代码根本不应该写。**
