# 封底估算：你的代码能扛多少 QPS？

我问一个程序员：“这个功能能扛多少 QPS？”

他说：“不知道，不压测怎么可能算出来？等上线以后再看吧。”

这就是大部分程序员的现状：他们认为封底估算是不可能的，所以会逃避这个事情，会交白卷。

但程序员就像一个土木工程师。他得知道自己造的是浮桥、坚固一些的石桥，还是能用百年的海湾大桥。

花的时间不一样，材料不一样，坚固程度也不一样。

**土木工程师不会说：“不知道，等造好再看看。”**

封底估算不要求 100% 准确。先判断功能能扛 10、1000 还是 10 万 QPS，就能排除数量级错误的方案。

**估算错了可以校准，交白卷永远不会变准。**

## 1. 找到木桶最短板

一个功能经过的内存、Redis、MySQL 和下游接口，都是木桶的一块木板。最短的那块，决定了这个功能能承载的最高 QPS。

```text
组件 QPS = 业务 QPS x 每次请求调用次数 x 实际发生比例
功能容量 = 所有组件可承载业务 QPS 中的最小值
```

第一次估算，可以先记住下面的数量级：

| 组件 | 估算起点 |
| --- | --- |
| 本机内存 | 简单查找可以从百万次/秒开始估算 |
| Redis | 官方示例约 7 万到 18 万次/秒；单个热 Key 到 1000 QPS 就应该重点检查 |
| MySQL | 4 核 16 GB 混合读写实测约 1 万到 4 万 QPS；单个业务到 100 QPS 就应该重点检查 |

1000 和 100 是业务审查线，不是物理上限。真实容量还要看硬件、数据量、SQL 和剩余负载。

## 2. 完整算一次

假设已有接口峰值是 3000 QPS，预计 20% 的请求会进入新功能：

```text
新功能峰值 = 3000 x 20% = 600 QPS
保留 3 倍冗余，容量目标 = 1800 QPS
```

每个请求查一次 Redis，缓存命中率是 95%：

```text
Redis = 1800 QPS
MySQL = 1800 x 5% = 90 QPS
```

正常时，Redis 热 Key 超过审查线，是短板。缓存失效时，MySQL 从 90 QPS 跳到 1800 QPS，又成了短板。

因此上线前就要考虑本机缓存、热点拆分、缓存预热、合并回源和限流。

用了缓存，还要顺手算内存：

```text
内存 = 数据条数 x 单条大小 x 额外开销 x 副本数
```

## 3. 最不能接受的是交白卷

资深程序员看代码，就应该知道哪块木板最短，哪里容易爆内存。把假设写下来，上线后用真实数据校准，下一次就会更准、更快。

**最不能接受的不是估算不准，而是认为不可能，所以什么都不做。**
