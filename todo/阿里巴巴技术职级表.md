# 阿里巴巴技术职级表（公开资料整理）

> 更新日期：2026-07-12
>
> 本文不是阿里巴巴官方发布的任职资格或晋升标准。职级编号、改革信息和历史 title 来自公开报道；“能力画像”是根据这些资料与公开招聘职责归纳的对标标准，不代表阿里内部原文。

## 先说结论

1. 阿里传统上使用专业序列 `P` 和管理序列 `M`。2020 年阿里取消了内部系统中的 `P` 序列展示，但当时不等于取消职级本身。
2. 2023 年公开报道显示，淘天集团讨论将原 `P4-P8` 改为 `14-28` 级，每个原 `P` 级对应三个数字级；原 `P8` 以上改走组织任命。
3. 当时淘天回应称方案仍在调研，后续公开材料对其是否集团范围完全落地存在口径差异。因此，不应把 `14-28` 直接称为“阿里巴巴集团当前官方完整职级表”。
4. 为了既反映改革又便于行业对标，下表同时保留报道中的数字级和传统 `P` 级。

| 14-16 | P4 | 初级工程师 | 掌握基础开发知识；在指导下按方案完成明确任务；能修复一般缺陷，补充测试和文档 | 单个需求、简单接口、工具脚本或存量模块维护 |
| 17-19 | P5 | 中级工程师 | 能独立拆解需求并交付完整模块；对代码质量、测试和上线结果负责；能定位常见线上问题 | 一个业务模块、小型服务或端上完整功能 |
| 20-22 | P6 | 高级工程师 | 能设计和交付复杂子系统；处理性能、稳定性和可扩展性问题；在项目中做关键技术决策；能辅导较低职级工程师 | 核心子系统、中型服务、技术改造或一条业务链路 |
| 23-25 | P7 | 专家 | 是某技术领域或业务域的负责人；主导中大型项目和跨团队协作；把复杂问题沉淀为平台、标准或方法论；对团队人才成长负责 | 业务域架构、共性平台、多系统技术方案或团队技术路线 |
| 26-28 | P8 | 高级专家 | 对一个大业务域或多个团队产生稳定影响；主导重大系统或技术平台演进；平衡技术、业务、成本和组织效率；建立技术梯队并影响跨组织决策 | 事业群级核心平台、重大基础设施或多团队技术战略 |
| 28 以上 | 原 P8 以上 | 报道称改走组织任命 | 不再仅按个人专业能力定级，更关注岗位价值、业务规模、团队规模与持续贡献 | 事业群或公司级技术负责人、核心业务负责人 |

`14-16` 等区间内的三个数字级，可理解为同一职级宽带内的成熟度递进：起步级关注能否在指导下完成，中间级关注能否稳定独立完成，高一级关注能否承担该宽带中更复杂的问题并带动他人。这一解读是为实际对标而做的归纳，不是公开的阿里内部逐级定义。

## 判定职级时应看什么

- **自主性：** 从按明确方案执行，到自主定义问题、选择解法并对结果负责。
- **复杂度：** 从单一模块，到复杂子系统，再到多系统、多团队和长周期技术治理。
- **影响范围：** 从个人交付，到项目、团队、业务域和事业群。
- **可复用贡献：** 是否把一次性解法沉淀为平台、标准、工具、方法论或人才梯队。
- **业务结果：** 高职级不只看技术难度，还要看稳定性、效率、成本、用户价值和持续性。
